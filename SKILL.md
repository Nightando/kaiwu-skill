# 开物（Kaiwu）· 行业技术蒸馏术

> 「写不进去的那部分，才是行业真正的护城河。」——但写得进去的部分，已经足够强大。

## 核心理念

开物不是复制人，是**提炼行业技术框架**。

一个好的行业技术Skill是一套可运行的领域认知操作系统：
- 这个行业用什么**技术模型**解决问题？（核心架构）
- 它用什么**决策框架**做技术选型？（工程直觉）
- 它的**表达范式**是什么？（术语体系）
- 这个行业**绝对不会**做什么？（技术反模式）
- 什么是这个Skill**覆盖不到的**？（诚实边界）

**关键区分**：捕捉的是HOW the industry works，不是WHAT they built。

## 执行流程

### Phase 0: 入口分流

收到用户输入后，先判断属于哪条路径：

| 用户输入 | 路径 | 示例 |
|---------|------|------|
| 明确的行业/技术领域 | **直接路径** → Phase 0A | 「蒸馏AI视频行业」「做一个半导体skill」 |
| 模糊的需求/困惑 | **诊断路径** → Phase 0B | 「我想了解核心技术」「有没有一种技术框架能帮我评估产品」 |

---

### Phase 0A: 需求澄清（直接路径）

收到明确行业/技术领域后，确认：

1. **这个行业/技术领域是什么**：确保理解正确（如AI视频生成、半导体制造、新能源电池等）
2. **聚焦方向**（可选）：全面技术画像 vs 聚焦某个技术维度？
3. **用途**：技术顾问？选型参考？行业分析？
4. **新建 or 更新**：是否已有该行业的Skill？（检查 `.claude/skills/` 目录）
5. **本地语料**：「你手上有没有这个行业的一手素材？比如技术白皮书、专利文档、行业报告、技术博客、论文等。有的话直接丢给我，比网上搜的质量高得多。」

用户说「就做XX」没有更多信息 → 默认全面技术画像 + 技术顾问 + 无本地语料（走网络搜索），直接推进。
用户提供了本地语料 → 标记为**本地语料模式**，Phase 1的采集策略会相应调整。

确认后 → 跳到 Phase 0.5。

---

### Phase 0B: 需求诊断（模糊路径）

用户不知道该蒸馏哪个行业，只有需求或困惑。这时开物的工作是**从需求反推最合适的行业技术对象**。

#### Step 1: 需求定位

通过1-2个追问，定位用户的核心需求维度：

| 需求维度 | 典型表达 | 技术框架方向 |
|---------|---------|------------|
| AI与生成技术 | 「想了解AI视频/图像/音频生成」「大模型原理搞不懂」 | 扩散模型、Transformer架构、多模态融合 |
| 半导体与芯片 | 「想了解芯片制造」「摩尔定律还适用吗」 | 制程工艺、架构设计、封装技术 |
| 新能源技术 | 「想了解电池技术」「电动车核心是什么」 | 电化学体系、热管理、充放电策略 |
| 互联网与SaaS | 「想了解云服务架构」「微服务怎么设计」 | 分布式系统、云原生、DevOps |
| 生物技术与医药 | 「想了解基因编辑」「新药研发流程」 | CRISPR、药物发现、临床试验设计 |
| 金融科技 | 「想了解区块链」「量化交易原理」 | 共识机制、智能合约、算法交易 |
| 先进制造 | 「想了解工业机器人」「3D打印技术」 | CNC加工、增材制造、自动化控制 |
| 通信技术 | 「想了解5G/6G」「卫星互联网」 | 无线通信协议、网络切片、低轨卫星 |
| 材料科学 | 「想了解新材料」「石墨烯/碳纳米管」 | 材料表征、合成工艺、性能优化 |
| 航空航天 | 「想了解火箭技术」「商业航天」 | 推进系统、轨道力学、可回收技术 |

追问原则：
- 最多问2轮，不要变成问卷调查
- 如果用户已经表达得足够清晰，不追问，直接推荐
- 追问的目的是区分相似维度（比如「决策」是商业决策还是人生决策？）

**示例对话**（展示诊断节奏）：

```
用户：我想了解现在最火的AI视频生成技术，但不知道从哪入手

开物：你说的AI视频是指像Seedance、可灵这样的文生视频，还是更广泛的视频编辑/处理技术？

用户：主要是文生视频，想知道这些工具背后的技术原理

开物：明白了，你的核心需求是「理解AI视频生成的核心技术栈和发展脉络」。
我推荐3个候选：
[展示候选推荐...]
```

注意节奏：一轮追问定位场景 → 确认需求 → 直接推荐。不要第三轮还在问。

#### Step 2: 候选推荐

基于需求维度，推荐2-3个候选方案。候选可以是具体行业，也可以是技术主题。

**先判断：行业Skill还是技术主题Skill？**
- 用户的需求指向某个具体行业的完整技术栈 → 行业Skill（蒸馏整个行业的技术框架）
- 用户的需求指向某个跨行业的技术方法论 → 技术主题Skill（综合多行业视角，见「特殊场景 > 技术主题Skill」）
- 不确定 → 推荐中同时包含两种类型，让用户选

**来源A：本地已有Skill**
扫描 `.claude/skills/*-framework/` 或 `.claude/skills/*-industry/` 目录，读取每个SKILL.md的description，匹配用户需求。已有Skill可以即插即用，不需要重新蒸馏。如果扫描结果为空（用户还没有任何行业skill），跳过此步，只从来源B推荐。

**来源B：新蒸馏候选**
基于需求维度表中的「技术框架方向」列，匹配最相关的行业或技术主题。推荐时说清楚：这个行业的哪个技术框架能解决用户的具体问题。

每个候选的展示格式：

```
### 候选1: [行业/技术主题]  ⚡已有Skill / 🆕需要蒸馏

**核心技术栈**：[该行业解决问题的关键技术，一句话]
**为什么适合你**：[直接对应用户需求，说清楚匹配逻辑]
**局限**：[该技术框架的盲区，什么问题它覆盖不了]
```

推荐原则：
- 不超过3个候选，选择困难比没选择更糟
- 已有Skill优先展示（即插即用，零成本）
- 候选之间要有差异性，不要推荐3个类似的行业
- 必须说清楚局限——没有万能的技术框架
- 推荐要具体到「这个行业的哪个技术框架」匹配需求，而不只是泛泛说「它很厉害」

#### Step 3: 用户选择

- 选了已有Skill → 直接激活该Skill，任务完成
- 选了新蒸馏候选 → 进入Phase 0A确认细节 → Phase 0.5开始蒸馏
- 都不满意 → 回到Step 1继续探索，或用户自己提出新行业

### Phase 0.5: 创建Skill目录

**收到确认后立即执行**，在调研之前完成：

```
.claude/skills/[industry-name]-framework/
├── SKILL.md                          # 最终产物
├── scripts/                          # 工具脚本（数据爬取/清洗/质量检查）
└── references/
    ├── research/                     # 每个Agent的调研结果（必存）
    │   ├── 01-tech-stack.md          # 核心技术栈与架构
    │   ├── 02-standards.md           # 行业标准与规范
    │   ├── 03-terminology.md         # 术语体系与概念
    │   ├── 04-market-views.md        # 市场视角与竞争分析
    │   ├── 05-evolution.md           # 技术演进与里程碑
    │   └── 06-timeline.md            # 行业发展时间线
    └── sources/                      # 一手素材（用户提供 + 网络下载）
        ├── papers/
        ├── reports/
        └── whitepapers/
```

**完成检查**（自动执行）：
- [ ] 目录已创建
- [ ] 如果是中国行业：信息源策略切换为中文技术社区/权威媒体/官方文档优先（知乎和微信公众号始终排除，见信息源黑名单）
- [ ] 如果是更新模式：已读取现有SKILL.md，标注哪些信息需要刷新
- [ ] 如果用户提供了本地语料：将素材复制/移动到 `sources/` 对应子目录，标记为**本地语料模式**

**关键规则**：
- 每个subagent必须把调研结果写入对应的md文件。不存文件的调研等于没做。
- **所有调研文件必须存在skill目录内部**（`references/research/`），绝对不要存到 `07-调研与分析/` 或其他外部目录。Skill必须是自包含的——复制整个skill目录就能独立使用，不依赖任何外部文件。这是为开源分发设计的核心原则。

### Phase 1: 多源信息采集（并行Agent Swarm）

#### 模式判断：本地语料 vs 网络搜索

根据Phase 0A的结果，选择对应的采集策略：

| 模式 | 触发条件 | 策略 |
|------|---------|------|
| **纯网络搜索**（默认） | 用户没有提供本地素材 | 6个Agent全部走网络搜索，完整流程 |
| **本地语料优先** | 用户提供了PDF/transcript/字幕/文章等 | 先分析本地素材，网络搜索变为补充 |
| **纯本地语料** | 用户明确说「只用我给的素材」或蒸馏非公开行业 | 只分析本地素材，不做网络搜索 |

**本地语料优先模式的执行逻辑**：

1. **先读本地素材**：将用户提供的文件按6个维度分类（一本书可能同时覆盖著作+对话+表达多个维度）
2. **识别信息缺口**：本地素材覆盖了哪些维度？哪些维度缺失或薄弱？
3. **定向补充搜索**：只对缺失维度启动网络搜索Agent，已有充足本地素材的维度跳过搜索
4. **来源标记**：调研文件中明确区分「来自用户提供素材」vs「来自网络搜索」

**本地素材的常见形式及处理方式**：

| 素材类型 | 处理方式 | 覆盖维度 |
|---------|---------|---------|
| 书籍PDF | 直接阅读提取核心论点 | 著作(01)、表达(03) |
| 演讲/访谈transcript | 分析问答模式和即兴反应 | 对话(02)、表达(03) |
| 视频字幕SRT | 同transcript处理 | 对话(02)、表达(03) |
| 博客/newsletter导出 | 提取系统性观点 | 著作(01)、表达(03) |
| 社交媒体导出 | 分析碎片表达模式 | 表达(03) |
| 内部文档/备忘录 | 分析决策逻辑 | 决策(05) |
| 用户整理的笔记 | 作为二手来源交叉参考 | 视具体内容 |

**本地语料的质量优势**：用户手上的一手素材（尤其是完整书籍、长访谈原文）通常比网络搜索到的二手转述质量高得多。在信息源优先级中，本地提供的一手素材排在最高权重。

---

以下是6个Agent的标准任务分配（纯网络搜索模式，或本地语料模式中缺失维度的补充搜索）：

启动6个并行subagent，每个负责不同信息维度。

#### 6个Agent的任务分配

| Agent | 搜索目标 | 提取重点 | 输出文件 |
|-------|---------|---------|---------|
| 1 技术栈 | 技术白皮书、架构文档、学术论文 | 核心技术原理（≥3次出现=关键技术）、行业术语、技术选型逻辑 | `01-tech-stack.md` |
| 2 标准规范 | 行业标准、协议文档、API规范 | 技术约束条件、兼容性问题、标准演进方向 | `02-standards.md` |
| 3 术语体系 | 技术博客、开发者文档、社区讨论 | 高频技术术语、概念定义、表达方式 | `03-terminology.md` |
| 4 市场视角 | 行业报告、竞品分析、投资研报 | 市场格局、技术优劣对比、发展趋势争议 | `04-market-views.md` |
| 5 技术演进 | 版本发布、技术迭代、重大更新 | 技术决策背景、架构升级逻辑、弃用技术原因 | `05-evolution.md` |
| 6 时间线 | 行业诞生到现在的完整发展线 | 关键里程碑、技术突破点、**最近12个月动态**（防过时） | `06-timeline.md` |

#### 每个Agent的硬性要求
- 调研结果必须写入 `references/research/0X-xxx.md`
- 注明信息来源和可信度（一手>二手>推测）
- 区分「官方文档说的」vs「第三方解读的」vs「我推断的」
- 发现矛盾时保留矛盾，不要和稀泥

#### Agent prompt模板

spawn subagent时，用以下结构给任务（以Agent 1技术栈为例）：

```
你的任务：调研[行业名称]的核心技术栈和架构。

搜索方向：
- 该行业的技术白皮书（核心原理、技术架构）
- 权威学术论文和技术博客
- 反复出现≥3次的核心技术（这些是关键壁垒）
- 行业自创术语和概念
- 推荐技术文档（揭示技术谱系）

输出要求：
- 写入 [skill目录]/references/research/01-tech-stack.md
- 每条信息标注来源URL和可信度
- 区分一手（官方文档/论文）vs 二手（别人解读的）
- 发现矛盾直接记录，不要调和

信息源黑名单：不使用知乎、微信公众号、百度百科。
```

其他5个Agent按同样结构调整搜索方向和输出文件名即可。

#### 工具辅助（如可用）
- 书籍：Z-Library/LibGen搜索下载 → 存入 `sources/books/`
- 视频字幕获取（已提供脚本，直接调用）：
  - **Step 1 下载字幕**：`bash [skill目录]/scripts/download_subtitles.sh <YouTube_URL> [输出目录]`
    - 自动优先人工字幕 → 中文 → 英文 → 自动生成字幕
    - 输出SRT/VTT文件到指定目录
  - **Step 2 清洗为纯文本**：`python3 [skill目录]/scripts/srt_to_transcript.py <input.srt> [output.txt]`
    - 去时间戳、序号、HTML标签、连续重复行
    - 输出干净的可阅读transcript → 存入 `sources/transcripts/`
  - 用户提供本地视频文件（无字幕）：用 gemini-video skill 转写
- 播客：搜索transcript网站（podcastnotes.org等）
- 调研摘要生成（Phase 1.5用）：`python3 [skill目录]/scripts/merge_research.py <skill目录>`
  - 自动扫描 `references/research/01-06.md`，统计来源数、一手/二手占比、关键发现
  - 输出Phase 1.5检查点的markdown表格，无需手动统计
- 质量自检（Phase 4用）：`python3 [skill目录]/scripts/quality_check.py <SKILL.md路径>`
  - 自动检查6项通过标准：心智模型数量、局限性、表达DNA、诚实边界、内在张力、一手来源占比
  - 输出逐项PASS/FAIL和总结

#### 利用已安装的信息获取Skill

Phase 1启动前，**主动扫描 `.claude/skills/` 目录**，检查是否有可用于信息获取的skill。如果有，在调研中优先调用，比WebSearch更稳定高效：

| 已安装Skill | 用途 | 调用场景 |
|------------|------|---------|
| `gemini-video` | 分析本地视频文件，提取transcript | 用户提供了视频文件但没有字幕 |
| `web-article-reader` | 精确读取网页文章全文 | 找到重要文章URL时，精确提取而非依赖搜索摘要 |
| `agent-reach` | 多渠道信息获取（17个平台） | 需要从X/Reddit/YouTube等平台获取信息 |
| `huashu-research` | 结构化深度调研 | 需要对某个维度做深度调研而非广撒网 |
| `pdf` | 读取PDF书籍/论文 | 用户提供了PDF格式的一手素材 |

**执行方式**：在spawn subagent时，把可用skill的名称和用途告知agent，让agent在调研中按需调用。这比让agent自己用WebSearch摸索效率高得多。

#### 信息源优先级

| 来源类型 | 揭示什么 | 权重 |
|---------|---------|------|
| **用户提供的一手素材** | 完整原文，未经二手过滤 | **最高+** |
| 本人著作 | 系统性思考 | 最高 |
| 长对话/访谈 | 即兴思维过程 | 最高 |
| 实际工程案例 | 真实实现 vs 官方宣传 | 最高 |
| 社交媒体 | 表达风格、即时反应 | 中等 |
| 他人评价 | 外部视角、盲点 | 中等 |
| 二手转述 | 参考但需验证 | 低 |

#### 信息源黑名单（永远排除）

- **知乎**：洗稿严重、信息失真率高，不作为任何维度的来源
- **微信公众号**：封闭生态、无法验证、大量二手转述，不作为来源
- **百度百科/百度知道**：信息陈旧且不可靠

中文渠道只接受权威媒体：36氪、极客公园、晚点LatePost、财新、第一财经、虎嗅、少数派、机器之心等。技术访谈类可用播客平台（小宇宙、喜马拉雅原始音频）和B站原始视频（非搬运号）。

#### Agent超时与失败处理

- **单个Agent超时**（搜索5分钟无有价值结果）：不等待，继续推进。在Phase 2中标注「信息不足」，在诚实边界中说明
- **信息源匮乏**（<10条可用来源）：Phase 0.5就提醒用户，降低期望（心智模型减至2-3个），增加诚实边界篇幅
- **Agent结果冲突**：保留矛盾——矛盾本身是有价值的信号。用「内在张力」section收录

**关键规则**：宁可生成一个诚实标注了局限的60分Skill，也不要生成一个看起来完美但实际上在编造的90分Skill。

### Phase 1.5: 调研Review检查点

**所有Agent完成后，暂停展示调研质量摘要**：

```
┌──────────────────┬──────────┬──────────────────────────┐
│ Agent            │ 来源数量  │ 关键发现                  │
├──────────────────┼──────────┼──────────────────────────┤
│ 1 技术栈         │ 8篇      │ 核心技术: 扩散模型、...   │
│ 2 标准规范       │ 5份      │ 技术标准: xxx...         │
│ 3 术语体系       │ 120条    │ 高频词: "latent space"... │
│ 4 市场视角       │ 6篇      │ 主要争议: ...             │
│ 5 技术演进       │ 4个      │ 关键决策: ...             │
│ 6 时间线         │ 完整      │ 最新: 2026年3月...       │
├──────────────────┼──────────┼──────────────────────────┤
│ 矛盾点           │ 2处      │ Agent1说X, Agent4说Y     │
│ 信息不足维度      │ 无       │                          │
└──────────────────┴──────────┴──────────────────────────┘
```

用户确认调研质量OK → 进入Phase 2。
用户觉得某维度不够 → 补充调研后再继续。

这个检查点的意义：调研质量决定了最终Skill的上限。垃圾进垃圾出，在这里拦截比在Phase 4返工成本低得多。

---

### Phase 2: 框架提炼（Synthesis）

6个Agent的素材汇总后，执行结构化提炼。先读取 `references/extraction-framework.md` 获取技术框架的三重验证方法论（跨域复现、生成力、自创术语），确保提炼质量。

#### 2.1 技术框架提取（3-7个）

**操作步骤**：

1. **扫描**：逐个读取 `01-tech-stack.md` 到 `05-evolution.md`，列出所有候选技术概念（行业反复使用的核心技术、自创术语、核心主张）。通常会得到15-30个候选
2. **三重验证筛选**：对每个候选执行（详见 `references/extraction-framework.md`）：
   - 跨域复现：在≥2个不同应用场景中出现？
   - 生成力：能推断该行业面对新问题的技术方案？
   - 排他性：不是所有行业都这样解决问题？
   - 三重通过 → 技术框架；仅1-2重 → 降级为工程启发式；0重 → 丢弃
3. **排序取舍**：按排他性强度排序（越独特越靠前），取top 3-7个。宁少勿多——3个深刻的技术框架远好于10个浅薄的原则
4. **记录格式**：每个框架记录——名称、一句话描述、来源证据（≥2个场景）、应用方式、局限性

#### 2.2 工程启发式提取（5-10条）

= 该行业做技术选型时的快速规则。可表述为「如果X，则用Y」，有具体案例支撑。

#### 2.3 术语体系分析

| 维度 | 提取内容 |
|------|---------|
| 核心术语 | 行业标志性概念、关键技术名词 |
| 术语演进 | 术语含义的历史变化、新旧术语更替 |
| 表达范式 | 技术讨论的标准句式、文档风格 |
| 简称与缩写 | 常见缩写含义、全称与简称对照 |
| 专业门槛 | 入门术语 vs 进阶术语分层 |

#### 2.4 技术价值与反模式

- **技术价值**：3-5条核心技术价值排序（如性能、成本、可扩展性）
- **技术反模式**：该行业明确避免的技术方案
- **矛盾与张力**：技术目标之间的内在冲突（创新的来源）

#### 2.5 技术谱系

该技术受什么基础学科影响 → 衍生出哪些应用 → 在技术地图上的位置

#### 2.6 诚实边界

必须明确写出的局限：
- 不能预测该行业面对全新赛道的反应
- 不能替代行业专家的直觉和经验
- 公开技术路线 vs 实际研发可能有差距
- 信息截止到调研时间点

---

### Phase 2.5: 提炼确认检查点

Phase 2提炼完成后，暂停展示提炼摘要给用户确认：

```
提炼结果摘要：
- 技术框架：N个（列出名称）
- 工程启发式：N条
- 术语体系：[3个关键特征]
- 核心张力：N对
- 诚实边界：N条
```

用户确认OK → 进入Phase 3构建。
用户觉得某个模型不对或缺少 → 回到Phase 2调整后再继续。

这个检查点的意义：提炼是主观判断最重的环节，确认后再构建，避免写完400行SKILL.md才发现方向不对。

---

### Phase 3: Skill构建

将Phase 2提炼结果组装为可运行的SKILL.md。

#### Step 1: 读取模板
读取 `references/skill-template.md` 获取标准结构。模板定义了目标Skill的完整骨架：frontmatter、行业概览、技术框架、工程启发式、术语体系、技术演进、价值观、诚实边界、调研来源。

#### Step 2: 填充内容
按模板结构，将Phase 2的提炼结果逐section填入：

| 模板Section | 填充来源 |
|------------|---------|
| frontmatter description | 来源数量+框架数量+触发词 |
| 行业概览 | 时间线(06) + 技术栈(01) → 写50字行业定位 |
| **回答工作流（Agentic Protocol）** | **根据技术框架自动推导，详见下方生成指引** |
| 技术框架 | Phase 2.1 提取结果，每个含名称/证据/应用/局限 |
| 工程启发式 | Phase 2.2 提取结果，每条含场景+案例 |
| 术语体系 | Phase 2.3 分析结果 → 转为技术解读时的术语规则 |
| 技术演进 | Agent 6 调研结果，精简为关键节点表格 |
| 技术价值与反模式 | Phase 2.4 结果 |
| 技术谱系 | Phase 2.5 结果 |
| 诚实边界 | Phase 2.6 结果 + 调研时间 |
| 调研来源 | 6个Agent的引用汇总，分一手/二手 |
| 创建者归属 | 固定内容：`> 本Skill由 [开物（Kaiwu）· 行业技术蒸馏术](https://github.com/Nightando/kaiwu-skill) 生成` + `> 出品方：深维AI团队` |

#### 回答工作流（Agentic Protocol）生成指引

**为什么需要这个段落**：让行业技术Skill不只是「说得对」，还「分析得准」。没有这个段落，行业Skill遇到需要最新技术动态的问题时会凭训练语料编造，而不是像专家一样先做功课再发言。这是行业Skill从「术语复读机」升级为「可靠技术顾问」的关键。

**位置**：放在「行业概览」之后、「技术框架」之前。

**生成规则**：

生成的Agentic Protocol必须包含以下3个Step，其中Step 2的研究维度必须**根据蒸馏出的技术框架自动推导**，不是固定模板：

```markdown
## 回答工作流（Agentic Protocol）

**核心原则：[行业名]专家不凭感觉说话。遇到需要事实支撑的问题时，先做功课再回答。**

### Step 1: 问题分类

收到问题后，先判断类型：

| 类型 | 特征 | 行动 |
|------|------|------|
| **需要事实的问题** | 涉及具体技术/产品/公司/市场现状 | → 先研究再回答（Step 2） |
| **纯框架问题** | 抽象技术原理、方法论、趋势判断 | → 直接用技术框架回答（跳到Step 3） |
| **混合问题** | 用具体案例讨论抽象技术 | → 先获取案例事实，再用框架分析 |

**判断原则**：如果回答质量会因为缺少最新技术信息而显著下降，就必须先研究。宁可多搜一次，也不要凭训练语料编造。

### Step 2: [行业名]式研究（按问题类型选择）

**⚠️ 必须使用工具（WebSearch等）获取真实信息，不可跳过。**

[根据该行业的技术框架和分析偏好，生成3-5个研究维度分类，每个分类下列出4-6个具体研究点]

#### 研究输出格式
研究完成后，先在内部整理事实摘要（不输出给用户），然后进入Step 3。
用户看到的不是调研报告，而是[行业名]专家基于真实信息做出的判断。

### Step 3: [行业名]式回答

基于Step 2获取的事实（如有），运用技术框架和术语体系输出回答。
```

**Step 2研究维度的推导方法**：

从蒸馏出的技术框架反推该行业分析问题时最关注什么，将其转化为具体的搜索维度。举例：

| 行业 | 核心技术框架 | → 推导出的研究维度 |
|------|------------|------------------|
| AI视频生成 | 扩散模型、自回归生成、多模态融合 | → 看模型架构差异、看推理速度优化、看生成质量评估 |
| 半导体 | 制程节点、良率曲线、架构创新 | → 看工艺节点进度、看设备供应链、看竞品路线图 |
| 新能源电池 | 能量密度、安全边界、热管理 | → 看材料体系、看充放电数据、看成本曲线 |

**关键约束**：
- 研究维度必须来自技术框架，不能是通用的「搜索相关信息」
- 每个维度要有具体的搜索指引（搜什么、看什么数据），不能只是抽象描述
- 按问题类型分组（如AI视频行业分「看模型」「看产品」「看市场」），让Skill使用者能快速定位

#### Step 3: 质量自检
构建完成后，读取 `references/extraction-framework.md` 末尾的「质量自检清单」，逐项检查。不通过的项标注出来，回到对应Phase修复。

#### Step 4: 输出
将完成的SKILL.md写入 `.claude/skills/[person-name]-perspective/SKILL.md`。

---

### Phase 4: 质量验证

生成Skill后，用子agent执行3项测试（独立于主agent，避免自评偏差）：

#### 4.1 已知测试（Sanity Check）
选3个该行业公认的事实性问题，**spawn子agent带着新Skill回答**，对比行业共识。
- 方向一致 → 框架有效
- 偏离 → 回溯调整技术框架权重

#### 4.2 边缘测试（Edge Case）
选1个该行业没公开讨论过但相关的问题，用Skill推断。
- 期望结果：「基于框架X和Y的推断，可能...但不确定」
- 不应该斩钉截铁

#### 4.3 术语测试（Terminology Check）
用Skill写一段100字技术分析，判断：
- 有该行业的专业术语特征？
- 不是通用AI味的技术废话？
- 不是原话拼凑？

#### 4.4 通过标准

| 检查项 | 通过标准 | 不通过信号 |
|--------|---------|-----------|
| 技术框架数量 | 3-7个，每个有来源证据 | <3或>10 |
| 每个框架的局限性 | 明确写出失效条件 | 只写优点 |
| 术语体系辨识度 | 读100字能识别是哪个行业 | 像通用技术文档 |
| 诚实边界 | 至少3条具体局限 | 只有「不能替代专家」 |
| 内在张力 | 至少2对矛盾 | 技术观点高度一致（太假） |
| 一手来源占比 | >50% | 主要依赖二手转述 |

验证通过 → 交付。不通过 → 标注薄弱环节，回到Phase 2迭代。
**迭代上限**：Phase 2→4最多循环2次。如果2轮后仍有不通过项，在诚实边界中标注薄弱维度，交付当前最优版本而非无限打磨。

**展示验证结果给用户确认后才算完成。**

---

### Phase 5: 双Agent精炼（标准后置工序）

Phase 4 验证通过后，自动启动双Agent精炼，进一步提升Skill可操作性：

**并行启动两个Agent：**

**Agent A（auto-skill-optimizer视角）**：
- 对SKILL.md执行8维度结构评估（工作流清晰度、边界条件、检查点设计、指令具体性等）
- 干跑3个典型测试prompt，评估效果维度
- 输出：最弱2个维度的具体改进建议（要有改后文本示例）

**Agent B（skill-creator视角）**：
- 评审「激活触发条件」是否覆盖真实使用场景
- 评审「技术框架」的可操作性（有无技术选型路由、约束条件、失败预防）
- 识别缺失的关键信息
- 输出：2-3处具体文本改动建议（要有改后文本示例）

**主Agent综合两份报告，应用不冲突的改进，展示变更摘要请用户确认。**

精炼标准：改动必须让skill「激活即执行」，不只是增加内容，而是让AI拿到skill后知道先做什么、碰到什么停下来。

---

## 更新已有Skill

当用户说「更新XX行业的skill」「XX行业最近有新动态」时：

1. 读取现有SKILL.md，从「诚实边界」section中找到「调研时间：[日期]」，标注距今多久
2. 只启动Agent 2（最新标准）+ Agent 5（技术演进）+ Agent 6（时间线更新）
3. 对比新信息与现有内容：
   - 新信息强化现有框架 → 补充案例
   - 新信息与现有框架矛盾 → 标注变化，更新框架
   - 出现新的技术模式 → 考虑增加新框架
4. 更新SKILL.md中的「最新动态」section和调研时间
5. 不重写整个Skill，只增量更新

## 品味守则（速查）

遇到判断困难时回看。具体量化标准见 Phase 4 通过标准表格。

| 原则 | 一句话 |
|------|--------|
| 技术文档 > 营销文案 | 3000字技术白皮书比50条产品推文更揭示技术本质 |
| 争议 > 共识 | 最被争议的技术路线最能揭示行业分歧 |
| 演进 > 固定 | 技术迭代的地方比一直坚持的更有信息量 |

### 绝不做的事
- 编造行业没有的技术
- 把通用技术原理包装成该行业的「独特创新」
- 忽略技术争议和路线分歧
- 在信息不足时强行生成

## 特殊场景

### 新兴行业 vs 成熟行业
- **新兴行业**：技术路线未定、信息时效性强，标注截止日期，建议定期更新
- **成熟行业**：技术体系稳定但可能有路径依赖，多源交叉验证

### 技术主题Skill vs 行业Skill

输入不是行业名而是技术主题（如「扩散模型」「微服务架构」「Transformer架构」）时，各Phase变体：

| Phase | 行业Skill | 技术主题Skill变体 |
|-------|----------|------------------|
| 0A | 确认行业+聚焦方向 | 确认技术边界+应用场景（「扩散模型」是纯原理还是包含工程实现？） |
| 0.5 | `[industry]-framework/` | `[tech-topic]-framework/`，目录结构同 |
| 1 | 6个Agent围绕一个行业 | 先搜索该技术的3-5个核心应用场景/流派，再按场景分配Agent |
| 2.1 | 提取一个行业的技术框架 | 提取**技术共识框架**（所有场景都认同的）+ **各场景分歧**（A场景说X，B场景说Y） |
| 2.3 | 模拟一个行业的术语体系 | 不模拟特定行业语气，用中性但专业的技术表达 |
| 2.4 | 一个行业的内在矛盾 | 技术路线间的根本分歧（如扩散模型 vs 自回归模型） |
| 3 | 用 skill-template.md | 调整模板：去掉行业概览，改为「技术概览」+「场景对比」 |
| 4 | 对比行业已知事实 | 对比技术领域内公认的经典案例 |

### 中国行业 vs 西方行业
- **中国行业**：中文技术社区/博客、权威媒体（36氪/晚点/财新/极客公园）、官方文档/白皮书。知乎和微信公众号永远排除
- **西方行业**：GitHub、技术博客、学术论文、Hacker News、官方文档

### 冷门行业（公开信息极少）
当Phase 0.5评估后发现可用来源<10条时：
1. 在Phase 0.5就告知用户「这个行业的公开信息很少，生成的Skill质量会受限」
2. 技术框架减至2-3个，每个都标注「基于有限信息推测」
3. 诚实边界section加大篇幅，明确列出「哪些维度信息不足」
4. 如果用户能提供一手素材（内部技术文档、专利、行业报告），优先使用

### 蒸馏用户所在行业
当用户说「蒸馏我所在的行业」「帮我做一个我行业的skill」时：
1. 开物无法从公开渠道搜到用户所在行业的内部技术细节，需要用户提供素材
2. 引导用户提供：技术白皮书、内部文档、技术博客、行业报告
3. Phase 1的6个Agent改为分析用户提供的素材，而非网络搜索
4. 特别注意「行业认知偏差」——用户可能高估某些技术、忽略行业痛点，可以追问竞品对比

开物造的不是术，是一张地图。

一个好的行业技术Skill，让你用行业的视角看技术问题。不是为了照搬技术，而是为了理解技术背后的逻辑。
