# 技术框架提炼方法论

> 从原始信息到可运行的行业技术框架，核心方法论

## 一、技术框架识别的三重验证

一个技术概念要被认定为「技术框架」而非「行业热词」，必须通过三重验证：

### 验证1: 跨域复现
同一个技术框架出现在该行业的至少2个不同应用场景。

例：AI视频生成的「扩散模型」——
- 在图像生成中应用（Stable Diffusion、Midjourney）
- 在视频生成中应用（SVD、Runway Gen-3）
- 在3D生成中应用（Zero-1-to-3、DreamFusion）
→ 跨3个领域复现 → 这是真正的技术框架

### 验证2: 有生成力
用这个框架可以推断该行业面对新问题的技术方案。

例：如果「扩散模型」是技术框架——
- 面对「长视频生成」→ 引入时序注意力机制
- 面对「实时生成」→ 引入蒸馏和量化加速
- 面对「可控生成」→ 引入ControlNet条件控制
→ 能生成新推断 → 这是真正的技术框架

### 验证3: 有排他性
不是所有技术领域都这样解决问题，这个框架体现了该行业的独特选择。

例：「扩散模型」vs「自回归模型」是AI生成领域的技术路线分歧
→ 有区分度 → 这是值得提炼的技术框架

**如果一个技术概念只通过了1-2重验证** → 降级为「工程启发式」而非「技术框架」
**如果一个技术概念0重验证都没过** → 可能只是行业热词，不纳入

## 二、术语体系的量化方法

### 2.1 术语指纹

从该行业的技术文档/论文/博客中随机抽取20个段落，统计：

| 维度 | 测量方式 |
|------|---------|
| 术语密度 | 专业术语数/总字数 |
| 缩写比例 | 缩写词数/总术语数 |
| 中英文混合度 | 中文术语vs英文术语比例 |
| 新旧术语更替 | 同一概念的不同表述版本 |
| 层级分布 | 入门术语vs进阶术语vs专家术语 |

### 2.2 表达范式标签

从以下维度打标：

```
学术化 ←→ 工程化
抽象 ←→ 具体
标准化 ←→ 口语化
长句技术描述 ←→ 短句结论
参数驱动 ←→ 经验驱动
文档导向 ←→ 代码导向
```

### 2.3 术语禁忌

- 该行业不用的过时术语 → 生成Skill时也不用
- 该行业明确区分的易混淆概念 → 必须准确区分
- 该行业的专属简称/缩写 → 首次出现时给全称

## 三、技术矛盾处理原则

技术矛盾是行业演进的核心动力，不是需要修复的Bug。

### 三种矛盾类型

1. **时间性矛盾**（技术路线演化）
   - 行业早期用技术A，现在主流用技术B
   - 处理：记录演化轨迹，标注「传统方案」「当前主流」「新兴方案」
   - 在Skill中以「当前主流」为主，但提及演化背景

2. **场景性矛盾**（不同场景不同选择）
   - 行业在场景X中主张技术A，在场景Y中主张技术B
   - 处理：分场景记录，不强求统一
   - 这恰恰是技术深度的来源

3. **本质性张力**（技术目标内在冲突）
   - 例：追求生成质量 vs 追求推理速度
   - 处理：明确记录为「核心张力」
   - 这通常是技术选型最关键的权衡点

### 错误的处理方式
- ❌ 选一边忽略另一边
- ❌ 编一个调和的解释
- ❌ 假装矛盾不存在

## 四、信息不足时的处理

| 情况 | 处理 |
|------|------|
| 某个维度公开信息很少 | 在Skill中标注「信息不足，此维度为推测」 |
| 只有二手信息 | 降低置信度，标注为「据[来源]报道」 |
| 信息相互矛盾且无法判断 | 并列呈现，让用户自行判断 |
| 行业刻意保密某技术 | 尊重边界，在Skill中注明「该技术的工程细节未公开」 |

## 五、行业Skill vs 技术主题Skill的区别

| 维度 | 行业Skill | 技术主题Skill |
|------|----------|--------------|
| 核心 | 一个行业的技术体系 | 一个技术方法的跨行业应用 |
| 技术框架来源 | 主要来自该行业内部 | 综合多行业视角 |
| 表达风格 | 行业专属术语体系 | 中性但专业的技术表达 |
| 矛盾处理 | 保留行业内部技术路线分歧 | 呈现跨行业应用差异 |
| 验证方式 | 对比行业内技术共识 | 对比技术原理的普适性 |

## 六、质量自检清单

生成Skill后，用以下问题自检：

### 技术框架
- [ ] 每个框架都有至少2个不同应用场景的证据？
- [ ] 框架数量在3-7个之间？（太少=太浅，太多=没提炼）
- [ ] 每个框架都有明确的应用场景和局限？
- [ ] 框架之间有技术张力但不矛盾？

### 术语体系
- [ ] 读起来有行业辨识度，不像通用技术文档？
- [ ] 术语使用准确，没有混淆概念？
- [ ] 抓住了行业专属表达而非表面模仿？

### 工程启发式
- [ ] 每条规则都有具体案例支撑？
- [ ] 可以被新情况触发（不只适用于原始案例）？
- [ ] 有明确的适用条件和失效边界？

### 诚实边界
- [ ] 明确写了做不到什么？
- [ ] 标注了信息源和调研时间？
- [ ] 承认了信息不足的维度？

### 整体
- [ ] 用该行业的技术视角看一个新问题，能得到有价值的判断？
- [ ] 不是技术文档的拼凑，而是框架的运行？
- [ ] 删掉行业名后，还能认出这是哪个行业的技术体系？
